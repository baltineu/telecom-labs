# Настройка RIP и немного теории
**Routing Information Protocol**

- Данный протокол динамической маршрутизации относится к дистанционно-векторным протоколам (каждый маршрутизатор делится своей таблицей маршрутизации с соседями);
- Метрика - количество промежуточных устройств ("хопов").

Работает на основе трех таймеров:

> - Регулярной рассылки (полной таблицы маршрутизации соседям);
> - Timeout'a (запускается при появлении записи в таблице маршрутизации, сбрасывается при получении сообщения-подтверждения этой записи);
> - Сборщика мусора (запускается, когда Timeout-таймер для данной записи истёк; записи выставляется бесконечная метрика (16), на срок действия таймера маршрут считается недостижимым; если не приходит подтверждение, запись истекает и запись удаляется).

**Команда** - по сути, тип сообщения, оно может быть либо запросом, либо ответом на запрос, либо сообщением регулярной рассылки.

**Идентификатор адресного пространства** - указывает тип адреса (IPv4).

**Тэг маршрута** - добавлено в v2 протокола, позволяет отличить внутренний (полученный по протоколу RIP) маршрут от внешнего (полученный от другого протокола, либо записанный статически или напрямую подключенный), что влияет на метрику.

**IP-адрес сети** - адрес сети назначения.

**Маска сети** - добавлено в v2 протокола, вносит поддержку бесклассовой (CIDR) адресации в протокол.

В ответе на запрос или в рассылке к заголовку прилагаются от 1 до 25 записей таблицы маршрутизации, записи добавляются в конец, каждая запись состоит из 5 строк (поля от идентификатора до метрики). В запросе присутствует только один вышеописанный заголовок.

Во второй версии протокола исправлены следующие недостатки первой версии:
- Широковещательная рассылка заменена на групповую;
- Отсуствие бесклассовой адресации.

Настройка данного протокола на роутерах 0, 1, 3 ограничена следующими командами:

`router rip` - начало настройки

`version 2` - обязательно выбрать версию с поддержкой бесклассовой адресации

`redistribute connected` - рассказывать соседям о своих напрямую подключенных сетях

`network 10.0.0.0` - запуск процесса RIP на интерфейсах с IP, попадающими в диапазон 10.0.0.0/8

`no auto-summary` - для передачи подсетей (отключение автоматического суммирования маршрутов)

На роутере 2 вдобавок необходимо ввести дополнительную команду:

`passive-interface se0/3/0` - отключение RIP на данном интерфейсе

# Сервер DHCP, DNS
Сначала немного теории.

В протокол сетевого уровня IP, служащий для маршрутизации в сетях и между сетями, встраивается (инкапсулируется) полезная нагрузка какого-либо (TCP, UDP) протокола транспортного уровня. В заголовке такого протокола обязательно присутствуют поля "Порт источника" и "Порт получателя". Эти порты необходимы, чтобы передавать данные определенному системному процессу (приложению). Связку IP + порт так же иногда называют socket. 

**UDP - User Datagram Protocol (протокол пользовательских датаграмм)**

Протокол без установления соединения. Используется там, где не так важна целостность данных, как скорость их передачи (потоковое видео/аудио).
Его заголовок включает в себя также поля "Длина сегмента" и "Контрольная сумма".

**DHCP - Dynamic Host Control Protocol.** Работает поверх UDP (порт компьютера 68, порт сервера DHCP 67). Необходим для автоматической настройки IP-адресации на конечных устройствах. Для этого применяется процесс DORA:

1) **D - Discover**

Src. IP: 0.0.0.0

Dest. IP: 255.255.255.255

Конечное устройство отправляет на широковещательный адрес (255.255.255.255) DHCP-запрос с UDP-порта 68 на UDP-порт 67. В самом DHCP-заголовке присутствует поле "Код операции", если его значение = 1, это означает, что данный пакет является запросом устройства. На данном этапе в заголовке DHCP отсутствуют IP-адреса сервера DHCP и конечного устройства, всё, что есть - это MAC-адрес устройства.

2) **O - Offer**

Src. IP: <IP-адрес сервера>

Dest. IP: 255.255.255.255

Когда какой-нибудь DHCP-сервер в данной сети получает запрос и собирается назначить устройству IP-адрес, первым делом необходимо убедиться, что данный адрес отсутствует в сети. Для этого сервер генерирует ARP-запрос с данным адресом. Если он не получает ответ на этот запрос, то отправляет конечному устройству DHCP-Offer, в котором указана необходимая информация (DNS-сервер, адрес шлюза, а также адрес сервера, клиентский IP-адрес и клиентский MAC). 

3) **R - Request**

Src. IP: 0.0.0.0

Dest. IP: 255.255.255.255

Далее устройство подтверждает намерение получить данный IP-адрес, отправляя ещё одну датаграмму, аналогичную Discover, но с указанным IP-адресом клиента.

4) **A - Acknowledge**

Src. IP: <IP-адрес сервера>

Dest. IP: 255.255.255.255

Сервер подтверждает назначение IP датаграммой, аналогичной Offer.  
  
**Настройка:**  
  
В рамках данной топологии настроим раздачу IP-адресов в сети 192.168.1.64/27 для интерфейсов fa0/3-10 (8 интерфейсов), для этого необходимо на Switch3 у этих портов настроить метку VLAN 2:

`int range fa0/3-10`

`switchport mode access`

`switchport access vlan 2`

Теперь при подключении к этим портам компьютеры оказываются в VLAN 2 и могут использовать DHCP.

# TCP

**Transmission Control Protocol** - протокол управления передачей

Протокол с установлением соединения. Используется там, где важна целостность данных. Перед передачей происходит трехэтапное рукопожатие (квитирование). После передачи каждого сегмента данных ожидается подтверждение.

**Номер последовательности (Sequence Number)** - порядковый номер первого байта данных, которые передаются в сегменте.

**Номер подтверждения (Acknowledge Number)** - порядковый номер первого байта данных, которые получатель ожидает в следующий раз.

**Флаги**

URG (Urgent) - поле "указатель важности" задействовано.

ACK (Acknowledgement) - флаг подтверждения.

PSH (Push) - передать данные конечному приложению.

RST (Reset) - сброс соединения.

SYN (Synchronization) - флаг установления соединения.

FIN (Finish) - флаг завершения соединения.

**Размер окна** - количество байт, которые готов передать отправитель.

**Указатель важности** - показывает смещение отностельно текущего номера последовательности на позицию последнего байта важной информации (которую необходимо передать, даже если передача запрещена).

Таймеры, на основе которых функционирует TCP:

> - Таймер повторной передачи  
Максимальное значение устанавливается на основе получения SYN/ACK при установлении соединения, далее усредняется. По истечении данного таймера происходит повторная передача сегмента. Сбрасывается при получении подтверждения.  
> - Таймер запросов  
Выставляется, когда получен сегмент с размером окна 0, если по истечении таймера не приходит сообщение с изменением размера окна, высылается запрос размера окна.  
> - Таймер контроля работоспособности (2 часа)  
Используется для долгих соединений.  
> - Таймер времени ожидания  
После закрытия соединения ресурсы некоторое время (пока не истечет таймер) не высвобождаются, чтобы запоздавшие сегмента не были опознаны как сегменты нового соединения.

# Почта и Web
  
**HTTP - HyperText Transfer Protocol**  
  
Порт сервера: 80  
Порт клиента: высокий (или динамический), > 1024  
  
Алгоритм работы:  

- Установление TCP-соединения;  
- Формирование и отправка запроса клиентом;  
- Формирование и отправка ответа;  
- Закрытие TCP-соединения.  
  
Промежуточными устройствами могут выступать:  
  
- **Прокси** - для защиты локальной сети и кэширования;  
- **Шлюз** - для проверки и проведения трафика из внешней сети во внутреннюю;  
- **Туннель** - для создания защищенного (шифрованного) соединения между двумя сетями через некую внешнюю сеть.

Типы соединений:  

- Непостоянные - это значит, чтобы скачать страничку, нужно (N+1) TCP-соединений - одно на саму страничку и ещё N по числу объектов, которые использует страница;
- Постоянные - все объекты страницы и страница будут получены в рамках одного TCP-соединения.  

Постоянные соединения бывают с *конвейеризацией* и без неё. Конвейеризация означает, что запросы на все объекты отправляются сразу, нет ожидания ответа на предыдущий запрос от сервера.  

В рамках данной лабораторной работы формат HTTP-пакета подробно не разбирается.  

**POP3 - Post Office Protocol**  
  
Порт сервера: 110  
Порт клиента: высокий (или динамический), > 1024  
  
Протокол получения (и только) почты).
  
Шифрования в данном протоколе, как и в SMTP, нет (даже пароли передаются незащищенно, если не использовать команду APOP, которая является расширением).  
Идея - попробовать реализовать защищенное соединение через OpenSSL (POP3DS/IMAPDS) и передать по нему почту в GNS3.  
  
В отличие от IMAP сначала скачивает всю почту себе, потом изменяет. Не дает менять почту сразу с нескольких устройств. Синхронизация только после завершения сессии (команда QUIT). Предпочтительный вариант для нестабильного или медленного соединения.
  
Далее приводятся режимы работы протокола и их команды.  
  
Авторизация:  
> - User  
> - Pass  
> - Quit  
  
Получение почты:  
> - Stat (получение информации об общем количестве сообщений);  
> - List [n] (получение информации об объеме какого-то конкретного сообщения под номером n);  
> - Retr [n] (вывод содержимого сообщения под номером n);  
> - Dele [n] (отметить сообщение как удаленное);  
> - Noop (клиент ещё на связи);  
> - Rset (снять отметки об удалении писем).  
  
Ответ об обновлении от сервера:  
> \+OK  
> \-ERR  
  
**SMTP - Single Mail Transfer Protocol**  
  
Порт сервера: 25  
Порт клиента: высокий (или динамический) > 1024  
  
Только отправка писем. Для хорошего шифрования необходимы надстройки.  
  
Алгоритм работы:  
> - Установка TCP-соединения;  
> - Авторизация: auth login  
> - mail from zzz@bmstu.ru  
> - rcpt to yyy@bmstu.ru  
> - Data `Hello! (тело письма)` enter+.+enter  
> - Quit (разрыв tcp-соединения)  
  
Варианты ответа сервера (первая цифра вид ответа, две другие конкретизируют):  
> - 2 _ _ - первая цифра 2 означает, что всё хорошо;  
> - 3 _ _ - идет процесс, необходимы действия;  
> - 5 _ _ - произошла ошибка.  
  
Примеры ответов:  
`250 OK` - команда отработана;  
`354 Start mail input, end with <CRLF>.<CRLF>` - напоминание, что для отправки письма необходимо нажать enter+.+enter;  
`350 No such user here` - указанный пользователь отсутствует на сервере.  
  
# Access lists (списки доступа)

Нужны для контроля прохождения трафика, блокировки трафика на входе и выходе с интерфейса.    
Стандартные списки доступа проверяют трафик только по адресу сети/хоста.  
`access-list <1..99> {permit | deny | remark} {address | any | host} [source-wildcard]`  
Первый аргумент команды - это номер списка от 1 до 99. Далее указывается действие, производимое над пакетом, удовлетворяющим условиям. И далее пишется само условие - IP-адрес источника пакета равен конкретному адресу или входит в диапазон, задаваемый связкой адреса и wildcard.  
Пример команды:  
`access-list 1 permit 192.168.1.0 0.0.0.255` - access-list с номером 1 запретит пакет от любого хоста из сети 192.168.1.0/24.  
  
Расширенные списки доступа можно настроить на проверку как параметров источника, так и получателя, проверку типа протокола транспортного уровня (udp/tcp) и его портов, проверку на icmp-протокол, на тип icmp-сообщения.  
`access-list <100..199> {permit | deny | remark} protocol {address | any | host} [source-wildcard] [operator operand] {address | any | host} [destination-wildcard] [operator operand]`  
Пример команды:
`access-list 101 permit icmp host 192.168.1.2 any`  
Запрещает любые icmp-сообщения от конкретного хоста (192.168.1.2) до любого другого хоста.  
  
В любом списке доступа проверка и применение правил происходит сверху вниз.  
В данном примере второе правило никогда не сработает:  
```
access-list 1 deny 192.168.1.0 0.0.0.255  
access-list 1 permit host 192.168.1.1  
```
  
Также в зависимости от производителя оборудования, в конец списка доступа добавляется неявное правило:  
`{permit | deny} any any`  
В случае Cisco в конце запрещено всё, поэтому становится осмысленным добавление правила `permit any any`.  
  
Чтобы применить нужный список на интерфейсе, настраивается следующая команда:  
`ip access-group <номер_списка> {in | out}`  
На одно направление можно применить только 1 список доступа, но один и тот же список доступа можно применить на разных интерфейсах и направлениях.
